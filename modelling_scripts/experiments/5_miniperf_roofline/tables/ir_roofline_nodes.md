# IR Roofline Node Summary — `ir_roofline_nodes.csv`

## Description

Per-node aggregate static analysis of LLVM IR generated by compiling Autoware C++ source files with the [miniperf](https://github.com/alexbatashev/miniperf) Clang plugin (`clang-19 -O3 -Xclang -fpass-plugin=miniperf_plugin.so`).

**14 of 15** target nodes were successfully compiled. The `motion_velocity_planner` could not be compiled due to unresolvable cross-package dependencies at the IR level.

The `lidar_centerpoint` node shows 0 FLOPs because its core computation is in CUDA/TensorRT (GPU), and only a thin CPU wrapper was compiled.

## Methodology

1. Each node's C++ source files were located in the Autoware workspace (`~/autoware/src`).
2. Compile flags (includes, defines) were extracted from the existing CMake build cache (`flags.make`).
3. Each `.cpp` was compiled to LLVM IR text (`.ll`) using `clang++-19 -S -emit-llvm -O3` with the miniperf Clang plugin loaded.
4. The plugin instruments loops at the LLVM IR level, inserting counters for floating-point operations, integer operations, and memory load/store bytes.
5. A Python script (`analyze_ir_roofline.py`) parses the `.ll` files to count operations per function and aggregate per node.

## Columns

| Column | Description |
|---|---|
| `node` | Autoware node short name |
| `files` | Number of `.ll` files successfully generated |
| `functions` | Number of LLVM IR functions with at least 1 operation |
| `fp_functions` | Number of functions containing floating-point ops |
| `total_flops` | Total static FP operation count (fadd, fmul, fdiv, fma×2, fcmp, etc.) |
| `total_int_ops` | Total static integer operation count (add, sub, mul, shl, etc.) |
| `total_mem_bytes` | Total memory transfer bytes (loads + stores, typed from IR) |
| `load_bytes` | Subset: load instruction bytes |
| `store_bytes` | Subset: store instruction bytes |
| `arithmetic_intensity` | FLOPs / total_mem_bytes (the roofline X-axis value) |

## Key Findings

- All nodes have **very low arithmetic intensity** (AI < 0.15), placing them firmly in the **memory-bound** regime of the roofline model.
- `multi_object_tracker` has the highest FLOP count (12,457) due to extensive Eigen matrix operations (Kalman filters, covariance transforms).
- `map_based_prediction` and `autonomous_emergency_braking` also show significant FP workloads from geometric computations (Boost.Geometry, coordinate transforms).
- Most nodes are **integer-dominated** — control flow, message handling, and data routing dominate over arithmetic.
- The `occupancy_grid_map_node` has the highest per-node AI (0.139) despite low absolute FLOPs, because its compiled source is a tight grid computation with few memory accesses.
