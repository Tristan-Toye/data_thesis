# CARET Visualization Graphs Guide

This document explains each graph type generated by the CARET visualization scripts.

---

## Callback Metrics

### Callback Frequency Timeseries
**File:** `callback_frequency_timeseries.html`

Shows how often each callback is invoked over time (Hz). 
- **High frequency**: Callbacks executing many times per second
- **Spikes/drops**: Indicate system load changes or timing issues
- **Use case**: Identify callbacks that fire too frequently or not often enough

### Callback Frequency Histogram
**File:** `callback_frequency_histogram.html`

Distribution of callback execution frequencies.
- **Narrow distribution**: Consistent timing
- **Wide distribution**: Variable callback rates
- **Multiple peaks**: Different callback groups with different rates

### Callback Period Timeseries
**File:** `callback_period_timeseries.html`

Time between consecutive executions of each callback (ms).
- **Stable line**: Consistent periodic execution
- **Jitter**: Variations indicate timing instability
- **Use case**: Identify timing consistency issues

### Callback Period Histogram
**File:** `callback_period_histogram.html`

Distribution of time periods between callback executions.
- **Sharp peak**: Deterministic timing
- **Long tail**: Occasional delays
- **Use case**: Analyze timing determinism

### Callback Latency Timeseries
**File:** `callback_latency_timeseries.html`

Execution duration of each callback over time (ms).
- **Increasing trend**: Possible memory leak or accumulation
- **Spikes**: Heavy processing or resource contention
- **Use case**: Monitor callback performance over time

### Callback Latency Histogram
**File:** `callback_latency_histogram.html`

Distribution of callback execution times.
- **Left-skewed**: Most executions are fast
- **Right tail**: Occasional slow executions
- **Use case**: Identify worst-case execution times

## Communication Metrics

### Communication Frequency Timeseries
**File:** `communication_frequency_timeseries.html`

Rate of message exchanges between nodes over time.
- **Stable rate**: Healthy communication
- **Drops**: Message loss or node delays

### Communication Period Timeseries
**File:** `communication_period_timeseries.html`

Time between consecutive messages on each topic.
- **Consistent**: Reliable periodic communication
- **Irregular**: Bursty or event-driven communication

### Communication Latency Timeseries
**File:** `communication_latency_timeseries.html`

Time from message publication to reception.
- **Low latency**: Efficient communication path
- **High latency**: Buffering or processing delays
- **Use case**: Identify communication bottlenecks

---

## Path Metrics

### Message Flow
**File:** `path_*_message_flow.html`

Visualization of message propagation through a data path.
- **Horizontal lines**: Time spent at each node
- **Diagonal lines**: Communication between nodes
- **Use case**: Trace end-to-end data flow timing

### Response Time Histogram
**File:** `path_*_response_time.html`

Distribution of end-to-end path response times.
- **Peak**: Typical response time
- **Tail**: Worst-case delays
- **Use case**: Evaluate path latency distribution

### Chain Latency
**File:** (Interactive visualization)

Breakdown of latency contributions along a path.
- **Node latency**: Time spent processing at each node
- **Communication latency**: Time in transit between nodes
- **Use case**: Identify which nodes contribute most to total latency

---

## Scheduling Visualization

### Callback Scheduling
**File:** `callback_scheduling.html`

Timeline showing when callbacks execute on different executors.
- **Color coding**: Different callbacks
- **Gaps**: Idle time
- **Overlaps**: Parallel execution
- **Use case**: Analyze scheduling behavior and concurrency

---

## Node Latency Rankings

### Cumulative Latency Chart
**File:** `cumulative_latency_chart.png`

Shows which nodes contribute most to total latency.
- **Bar chart**: Latency per node (sorted highest first)
- **Cumulative line**: Running total of latency contribution
- **80% line**: Identifies nodes responsible for 80% of latency
- **Use case**: Prioritize optimization efforts

### Node Latency Table
**File:** `node_latency_table.html`

Interactive table with:
- Node name
- Latency in milliseconds
- Percentage of total latency
- Percentage of longest path
- Whether node is in longest path

**Color coding:**
- 🔴 Red: >10% of total latency (high priority)
- 🟡 Yellow: 5-10% (medium priority)
- 🟢 Green: <5% (low priority)

## Interpretation Guidelines

### Identifying Bottlenecks
1. Check **cumulative latency chart** - top nodes need attention
2. Review **callback latency histogram** for worst-case times
3. Examine **message flow** for path-specific delays

### Timing Analysis
1. Use **period histograms** for determinism analysis
2. Check **frequency timeseries** for rate stability
3. Analyze **scheduling** for executor utilization

### Optimization Priorities
1. Focus on nodes with >10% latency contribution
2. Address spikes in latency timeseries
3. Improve communication paths with high latency
