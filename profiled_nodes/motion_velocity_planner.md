# Motion Velocity Planner Node

## Node Name
`/planning/scenario_planning/lane_driving/motion_planning/motion_velocity_planner`

## Algorithms
- **Velocity Optimization**: Multi-constraint velocity profile generation
- **Obstacle Stop Planning**: Time-to-collision based deceleration
- **Curvature-Based Speed Limits**: Lateral acceleration constraints
- **Traffic Rule Compliance**: Stop lines, traffic lights, crosswalks

## Parameters

### Performance Impacting

- **Planning modules enabled** (various bools):
  - **Description**: Enable/disable specific velocity planning modules
  - **Default**: Various (crosswalk, intersection, obstacle, etc.)
  - **Impact**: Each active module adds processing overhead. Typical: 5-10 modules active, ~2-5ms each

- **`slow_down_margin`** (double):
  - **Description**: Distance margin for obstacle slow-down
  - **Default**: `5.0` meters
  - **Impact**: Larger margins trigger earlier deceleration, more conservative behavior

- **`stop_margin`** (double):
  - **Description**: Distance before stop point
  - **Default**: `0.0` meters
  - **Impact**: Safety buffer, affects final stop position accuracy

- **`max_lateral_accel`** (double):
  - **Description**: Maximum lateral acceleration for curves
  - **Default**: `2.0` m/s²
  - **Impact**: Directly determines speed limits on curves. Lower values → slower cornering

### Other Parameters

- **`crosswalk_slow_down_velocity`** (double):
  - **Description**: Speed limit when approaching crosswalks
  - **Default**: `3.0` m/s (10.8 km/h)
  - **Purpose**: Pedestrian safety near crossings

- **`detection_area_margin`** (double):
  - **Description**: Expansion of detection zones
  - **Default**: `1.0` meters
  - **Purpose**: Conservative obstacle detection

- **`hold_stop_margin_distance`** (double):
  - **Description**: Distance to maintain stopped state
  - **Default**: `0.5` meters
  - **Purpose**: Prevents oscillation near stop lines

- **`intersection_velocity`** (double):
  - **Description**: Maximum speed through intersections
  - **Default**: `6.0` m/s (21.6 km/h)
  - **Purpose**: Safe intersection traversal

- **`blind_spot_detection_distance`** (double):
  - **Description**: Blind spot monitoring range
  - **Default**: `50.0` meters
  - **Purpose**: Lane change safety checks

## Explanation

### High Level

The Motion Velocity Planner assigns velocity values to the path generated by the behavior planner, creating a complete trajectory (path + velocity profile). It considers multiple constraints simultaneously: curvature limits, obstacle proximity, traffic rules (stop lines, traffic lights, crosswalks), and comfortable deceleration limits. Each constraint proposes a velocity limit, and the planner takes the minimum at each point, ensuring all constraints are satisfied.

The node operates through multiple sub-modules, each handling specific scenarios (obstacles, intersections, crosswalks, etc.). Each module scans the path for relevant constraints, computes required velocities, and inserts stop or slow-down points. The final velocity profile is smooth, safe, and respects all regulatory and physical constraints.

### Model

#### Velocity Constraint Framework

**Multi-Constraint Optimization:**

At each waypoint i, velocity v_i must satisfy:
```
v_i ≤ min(v_curve, v_obstacle, v_traffic, v_comfort, v_max)
```

Where:
- v_curve: curvature-based limit
- v_obstacle: obstacle-based limit
- v_traffic: traffic rule limit
- v_comfort: comfort-based limit
- v_max: vehicle maximum speed

#### Curvature-Based Speed Limit

**Lateral Acceleration Constraint:**

For curve with radius R:
```
a_lat = v² / R ≤ a_lat_max
```

Solving for velocity:
```
v_curve = √(a_lat_max · R)
```

**Curvature Computation:**

For path points {p₁, p₂, ..., pₙ}:

Curvature at point i:
```
κ_i = |dθ/ds|_i ≈ (θᵢ₊₁ - θᵢ) / Δs
```

Where θ is heading angle, Δs is arc length increment.

Radius:
```
R_i = 1 / κ_i
```

**Velocity Limit:**
```
v_i = min(v_{i-1}, √(a_lat_max / κ_i))
```

#### Obstacle-Based Velocity Planning

**Time-to-Collision (TTC):**

For obstacle at distance d with relative velocity v_rel:
```
TTC = d / v_rel
```

**Required Deceleration:**

To stop before obstacle:
```
a_required = -v² / (2·d)
```

**Safe Velocity:**

Given safe following distance d_safe and maximum deceleration a_max:
```
v_safe = √(2 · a_max · (d - d_safe))
```

**RSS Safe Distance:**

```
d_safe = v·t_react + v²/(2·a_ego) - v_obs²/(2·a_obs) + margin
```

**Velocity Profile:**

For obstacle at distance d_obs:
```
if d < d_critical:
  v_obstacle = 0 (emergency stop)
elif d < d_safe + slow_down_margin:
  v_obstacle = v_safe
else:
  v_obstacle = v_max (no limit)
```

#### Traffic Rule Constraints

**Stop Line Planning:**

For stop line at distance d_stop:

**Deceleration Profile:**

Required deceleration:
```
a = -v²_current / (2·d_stop)
```

If a < a_comfort:
```
Use smooth deceleration: a(s) = a_comfort
```

Else (emergency):
```
Use maximum deceleration: a(s) = a_max
```

**Velocity at distance s from stop:**
```
v(s) = √(2·a·(d_stop - s))
```

**Traffic Light Logic:**

```
if light == RED or light == YELLOW:
  Insert stop point at stop line
elif light == GREEN:
  No velocity constraint
elif light == UNKNOWN:
  Cautious approach: v = v_cautious
```

**Crosswalk Behavior:**

```
if pedestrian detected in crosswalk:
  Insert stop before crosswalk
elif approaching crosswalk:
  v = min(v_current, crosswalk_slow_down_velocity)
```

#### Comfort-Based Smoothing

**Jerk Limits:**

Acceleration change rate:
```
|da/dt| ≤ jerk_max
```

Typical: jerk_max = 1.0 m/s³

**Smooth Deceleration:**

For stopping at distance d:

```
Phase 1: Jerk-limited acceleration decrease
  a(t) = a₀ - jerk_max · t
  until a = a_target

Phase 2: Constant deceleration
  a(t) = a_target
  until v = v_final

Phase 3: Jerk-limited acceleration increase (if needed)
  a(t) = a_target + jerk_max · t
  until a = 0
```

**S-Curve Velocity Profile:**

```
v(t) = v₀ + ∫₀ᵗ a(τ) dτ
```

Produces smooth velocity changes without abrupt jerks.

#### Module Processing Pipeline

**Sequential Module Execution:**

```
For each enabled module M:
  1. Scan path for relevant features (obstacles, stop lines, etc.)
  2. Compute velocity constraints
  3. Insert stop points if needed
  4. Update velocity profile
```

**Module Examples:**

1. **Obstacle Stop Module**:
   - Detects obstacles in path
   - Computes TTC and safe distances
   - Sets stop or slow-down points

2. **Intersection Module**:
   - Checks intersection entry conditions
   - Enforces speed limits
   - Handles turn restrictions

3. **Crosswalk Module**:
   - Monitors pedestrian presence
   - Enforces slow-down zones
   - Triggers stops for crossing pedestrians

4. **Traffic Light Module**:
   - Reads traffic light states
   - Inserts stops at red lights
   - Manages yellow light decisions

**Velocity Merging:**

After all modules process:
```
For each waypoint i:
  v_final[i] = min(v_module1[i], v_module2[i], ..., v_moduleN[i])
```

Takes most restrictive constraint.

### Complexity

**Time Complexity:**

**Per Module:**

**Feature Detection:**
- Scan path for relevant features: O(N)
  - N = number of waypoints (typically 150)
- Spatial queries (nearby obstacles): O(K log K)
  - K = number of objects

**Velocity Computation:**
- Per waypoint constraint evaluation: O(1)
- All waypoints: O(N)

**Stop Point Insertion:**
- Binary search for insertion: O(log N)
- Array insertion: O(N) worst case

**Per Module Total:** O(N + K log K)

**All Modules:**

M active modules:
```
T_total = M · (N + K log K)
```

For typical values:
- M = 8 modules
- N = 150 waypoints
- K = 30 objects
- Time: 5-15 ms

**Velocity Smoothing:**
- Forward pass: O(N)
- Backward pass (propagation): O(N)
- Total: O(N)

**Total Per Cycle:**
```
T_cycle = O(M · (N + K log K) + N)
        ≈ O(M · N) for M << K
```

**Space Complexity:**

**Velocity Profile:**
- N waypoints × velocity: N × 4 bytes
- 150 × 4 = 600 bytes

**Module State:**
- M modules × state: M × 200 bytes ≈ 1.6 KB

**Obstacle Data:**
- K objects × metadata: K × 100 bytes = 3 KB

**Total:** ~5-10 KB (minimal)

**Performance Bottlenecks:**

1. **Multiple Module Overhead**:
   - Each module iterates through path
   - Redundant computations if features don't exist
   - Mitigation: Early termination, feature indexing

2. **Obstacle Queries**:
   - Spatial lookups can be expensive
   - Repeated for each waypoint in some modules
   - Mitigation: Spatial caching, batch queries

3. **Stop Point Insertion**:
   - Array manipulation for each stop
   - Can fragment path representation
   - Mitigation: Preallocate space, use linked structures

4. **Velocity Propagation**:
   - Backward pass to ensure kinematic feasibility
   - Must check each waypoint
   - Mitigation: Sparse representation, change detection

5. **Module Activation**:
   - Some modules always active even when unnecessary
   - Fixed overhead regardless of scenario
   - Mitigation: Conditional activation based on context

**Parameter Trade-offs:**

- **Module Configuration**:
  - More modules: Comprehensive handling, higher latency
  - Fewer modules: Faster, may miss constraints
  - Optimal: Enable only necessary modules for operational design domain

- **`slow_down_margin`**:
  - Small (2-3m): Less conservative, later braking
  - Large (7-10m): Very conservative, early braking
  - Optimal: 5m balances safety and efficiency

- **`max_lateral_accel`**:
  - Low (1.0 m/s²): Very comfortable, slow cornering
  - High (3.0 m/s²): Faster cornering, less comfortable
  - Optimal: 2.0 m/s² matches typical passenger comfort expectations

- **`stop_margin`**:
  - Zero: Precise stops, risk of overshoot
  - Large (1m): Conservative, may stop too early
  - Optimal: 0-0.5m depending on control system accuracy

- **`crosswalk_slow_down_velocity`**:
  - Low (2 m/s): Very cautious, may impede traffic
  - High (5 m/s): Less conservative, faster flow
  - Optimal: 3-4 m/s balances pedestrian safety and traffic flow

- **`intersection_velocity`**:
  - Low (3 m/s): Very safe, slow intersections
  - High (10 m/s): Faster throughput, less margin
  - Optimal: 6-8 m/s for typical urban intersections

## Summary

The Motion Velocity Planner generates velocity profiles for paths by considering multiple constraints including curvature limits, obstacle proximity, traffic rules, and comfort bounds. It uses modular sub-planners for different scenarios (obstacles, intersections, crosswalks), merges their constraints by taking the minimum velocity at each point, and ensures smooth, safe, and rule-compliant trajectories for autonomous driving.
